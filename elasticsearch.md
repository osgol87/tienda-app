# ¿Cómo funciona el motor de búsqueda?

## Contenido

### Elasticsearch

El catálogo de productos vive en un clúster de Elasticsearch 7.10.2 gestionado por un tercero, Bonsai Cloud, al que `ProductServiceElastic` se conecta por SSL con autenticación básica. La configuración de ese cliente (`ElasticsearchConfig.java`) revela un detalle que no aparece en ningún documento del proyecto pero que explica por qué el servicio funciona: la clase extiende `ElasticsearchConfiguration`, la base del cliente Java moderno de Elastic (pensado para servidores 8.x), mientras que el servidor real de Bonsai sigue en la rama 7.10. Desde la versión 7.14, Elastic exige que el servidor responda con un encabezado `X-Elastic-Product: Elasticsearch` para que el cliente confirme que no está hablando con un fork o una imitación de la API; un servidor 7.10 no manda ese encabezado. La solución que adoptó el proyecto no fue alinear versiones, sino interceptar cada respuesta HTTP y añadir el encabezado a mano antes de que el cliente la procese. Funciona, pero es un parche sobre una incompatibilidad de versiones, no una corrección de la incompatibilidad en sí: cualquier actualización de Spring Data Elasticsearch que cambie cómo valida ese encabezado deja el servicio sin poder conectarse, sin que el código de negocio haya cambiado en nada.

El mismo archivo agrega un segundo interceptor que fuerza el encabezado `Content-Type: application/json` en cada petición saliente, quitando cualquier valor previo y reemplazándolo. Es la clase de ajuste que normalmente aparece cuando un proxy o un balanceador intermedio —en este caso, la capa de Bonsai— altera las cabeceras antes de que lleguen al clúster.

### Índices

Existe un único índice, `products`, declarado con `@Document(indexName = "products", createIndex = true)` sobre la entidad `Product`. La anotación `createIndex = true` significa que Spring Data Elasticsearch crea el índice automáticamente al arrancar el servicio, con la asignación de tipos (*mapping*) que se infiere de las anotaciones `@Field` de la clase Java. No hay un archivo de configuración de índice aparte, ni ajustes explícitos de número de *shards* o réplicas, ni un analizador personalizado: todo lo que existe hoy es lo que Spring Boot decide por defecto la primera vez que el servicio arranca contra un índice vacío.

De los ocho campos del documento, solo tres —`name`, `brand`, `category`— se declaran como `text`, el tipo que Elasticsearch analiza y tokeniza para poder buscarlo. Los demás (`short_description`, `long_description`, `image_url`) se marcan como `object` con `enabled = false`: se guardan, pero no se analizan ni se pueden usar en una búsqueda. Es una decisión razonable para no inflar el índice con texto largo que nadie va a consultar por palabra clave, aunque tiene una consecuencia que vale la pena anotar: si en algún momento se quisiera permitir buscar "tenis para correr en trail" dentro de la descripción larga de un producto, no bastaría con cambiar una consulta —habría que reindexar todo el catálogo con un mapeo nuevo, porque un campo marcado como no analizado no se puede convertir en analizado sin recrear el documento.

Tampoco hay un analizador de idioma configurado. Por defecto, Elasticsearch tokeniza `name`, `brand` y `category` con el analizador `standard`, pensado para inglés, que no hace *folding* de acentos ni de la ñ. Para una tienda que opera en México, donde una proporción considerable de las búsquedas se escribe desde el teclado de un celular —donde el autocorrector mexicano típicamente omite tildes o las coloca de forma inconsistente— esto importa: una marca o categoría almacenada como "Edición" no calzará automáticamente con una búsqueda tecleada como "edicion" a menos que el usuario escriba el acento exacto. El parámetro `fuzziness: 1` de la consulta (que se revisa en la siguiente sección) tolera un solo carácter de diferencia, pero una tilde faltante ya consume ese margen completo, dejando cero tolerancia para un error de tipeo real en la misma palabra.

### Flujo de búsqueda

Una petición típica entra por el Gateway como `GET /productservice/products?search=nike`, se enruta a `ProductsController.getProducts`, y de ahí baja a `ProductServiceImpl.getProducts`. Ahí ocurre una bifurcación simple pero con una consecuencia operativa concreta: si el parámetro `search` trae texto, el servicio llama a `ProductRepository.searchProducts`, que ejecuta la consulta Elasticsearch definida a mano con `@Query`:

```
multi_match { query: "?0", type: "bool_prefix", fuzziness: "1", fields: ["name","brand","category"] }
```

Si `search` viene vacío —que es el caso de la pantalla inicial del catálogo, cuando el cliente todavía no ha escrito nada— el servicio no usa Elasticsearch como buscador en absoluto: llama a `findAll()` y devuelve el índice completo en una sola respuesta. No hay parámetro de página, ni `size`, ni `from` en ningún punto de la cadena —controlador, servicio o repositorio—. Con un catálogo de un centenar de pares de tenis esto no se nota; en cuanto la tienda crezca a varios miles de productos, cargar el catálogo inicial va a significar traer todo el índice de un jalón cada vez que alguien abre la página principal.

El tipo `bool_prefix` de `multi_match` está pensado para autocompletado: coincide con el prefijo del último término que el usuario está escribiendo, útil si el frontend dispara la búsqueda en cada tecla. Lo que no hace esta consulta es dar más peso a un campo que a otro: `name`, `brand` y `category` entran con el mismo valor en la lista de campos, sin la sintaxis de refuerzo que Elasticsearch permite (`name^3`, por ejemplo). Buscar "correr" puede traer primero un producto cuya *categoría* es "correr" aunque el nombre no tenga nada que ver, empatado en relevancia con un producto que se llama literalmente "Zapatilla para Correr". Elasticsearch ordena por `_score`, pero aquí el `_score` no refleja ninguna prioridad de negocio sobre qué campo importa más al buscar un tenis.

## Análisis

El motor cumple lo que se le pidió en el caso de uso principal: tolerar errores de tipeo y ofrecer coincidencias parciales, algo que la versión anterior basada en `LIKE` sobre PostgreSQL no ofrecía. La decisión de no analizar las descripciones largas también es sensata para el tamaño actual del proyecto. Dicho esto, el motor tiene varios puntos que un catálogo real —no uno de demostración académica— no podría sostener sin ajustes:

La ausencia de paginación en el listado general es el más urgente, porque no depende de cuánto crezca el catálogo en el futuro: ya hoy, cada carga de la página principal del catálogo trae el índice entero, sin límite. El *health check* en `/products/health` tampoco verifica nada relacionado con Elasticsearch: devuelve un texto fijo ("Service is up and running") sin comprobar si el clúster de Bonsai responde, así que un monitor que consulte ese endpoint puede reportar el servicio como saludable mientras la búsqueda real está caída por un problema de red o de credenciales.

La falta de un analizador que normalice acentos y la ñ es un vacío que, en un mercado como el mexicano, no es un detalle cosmético: afecta directamente cuántas búsquedas legítimas encuentran resultado. Y la validación de `ProductServiceImpl.createProduct` y `updateProduct` exige los ocho campos completos en cada solicitud —incluida una actualización—, así que no existe manera de cambiar solo el precio de un producto sin reenviar también su nombre, marca, categoría y ambas descripciones; cualquier campo vacío hace que la petición entera se rechace con una excepción genérica (`InvalidProductRequest`), sin indicar cuál de los ocho campos faltó.

## Transición

El cambio de fondo, ya descrito desde la base de datos en el informe anterior, se ve aquí desde el ángulo de la búsqueda: pasar de un `LIKE` sobre columnas de texto a una consulta `multi_match` con tolerancia a errores no es solo cambiar de motor, es asumir un conjunto de responsabilidades que una base relacional nunca exigió. Un índice de Elasticsearch tiene un ciclo de vida propio —el mapeo de un campo no se puede alterar sin reindexar, el tamaño del índice crece con cada documento indexado como texto analizado, la relevancia de una búsqueda depende de decisiones explícitas de boosteo que aquí todavía no existen—. Nada de eso tenía equivalente en una consulta `LIKE '%termino%'`.

Lo que falta para que esta transición se pueda dar por terminada, más que por iniciada, es: paginar el listado, definir un analizador con soporte de español (o al menos con un filtro de *asciifolding* para tildes y ñ), dar peso explícito al campo `name` sobre `category` y `brand` en la consulta, y sustituir el interceptor que falsifica el encabezado de identificación del servidor por una alineación real de versiones entre el cliente y el clúster de Bonsai —o por un plan de Bonsai que ya corra Elasticsearch 8.x—. Ninguno de estos puntos es visible para un usuario que prueba el buscador con un catálogo pequeño y sin acentos, que es probablemente la razón por la que ninguno se ha corregido todavía.
